AEL: Docker, Dual-Stack, and “Skinny but Tall” Guidance

Memorialized guidance on using Docker, Python, and .NET while building the Adaptive Execution Loop engine end-to-end.

Core idea: prioritize an end-to-end vertical slice of the engine before expanding breadth. Build the engine “skinny but tall.”

1. Validate the Direction: Docker + Dual Stack

The recommended direction is sound:

  • Python in Docker as the reference engine for fast iteration and portability.
  • .NET locally as a parallel validation track and possible long-term enterprise implementation.

This arrangement helps prove that the underlying AEL engine is genuinely host-language agnostic rather than accidentally dependent on a specific runtime or ecosystem.

Architectural benefit: this approach protects the abstraction. It reduces the risk of building “a Python tool” instead of building an actual engine model.

2. What “Skinny but Tall” Means in AEL

“Skinny but tall” means building one narrow but complete vertical slice through the system. The objective is to make the engine run end-to-end before broadening the ontology, policy catalog, or runtime sophistication.

Wrong approach: wide but shallow

  • Too many node types
  • Too many policies
  • No actual execution loop
  • No real decisioning

Right approach: skinny but tall

  • Minimal YAML
  • Minimal graph
  • One policy
  • One decision rule
  • One complete loop
YAML → Graph → Policy → Decision → Action → Update → Log

If this works, the architecture is real. If it does not work, adding more concepts only increases confusion.

3. First Vertical Slice

The first slice should be aggressively constrained:

  • Node types: Requirement, CodeUnit
  • Edge type: realizes
  • Policy: “No orphaned code”
  • Loop behavior: evaluate → decide → print or log

That is enough to prove that the engine can:

  • read declared state,
  • build a graph,
  • evaluate a policy,
  • produce a decision,
  • emit a useful result.

4. Example Minimal Flow

1. YAML contains:
   - one Requirement
   - one CodeUnit
   - no realizes trace edge

2. Engine runs.

3. Policy evaluation detects:
   - orphaned code

4. Decision logic produces:
   - violation
   - create trace-reconciliation task

5. Output is printed or logged.

This is intentionally small. It is sufficient to validate the architecture.

5. Minimal Docker Strategy

The Docker setup should remain intentionally simple. It is there to guarantee portability and repeatability, not to introduce orchestration complexity prematurely.

FROM python:3.11-slim

WORKDIR /app
COPY . .

RUN pip install pyyaml

CMD ["python", "main.py"]

Suggested execution flow:

docker build -t ael-engine .
docker run -v $(pwd)/data:/app/data ael-engine

The goal at this stage is not scale. The goal is a stable, reproducible execution environment.

6. Parallel .NET Track

The .NET track should not become a second independent engine effort. Instead, it should function as a controlled validation path.

  • Python remains the reference implementation for rapid experimentation.
  • .NET mirrors the same YAML inputs and expected decisions at a minimal level.

This parallel path tests whether the engine model is portable across runtimes. That is the real architectural question.

Important constraint: do not let .NET become a competing conceptual model. It is a validation track, not a conceptual fork.

7. Critical Discipline Rule

No logic should live outside the engine model.

In practice this means:

  • no hidden rules hard-coded in one runtime but not the other,
  • no silent assumptions buried in a helper class,
  • no behavior that cannot be explained in terms of graph state, policy, DSL, or explicit engine logic.

The engine must remain understandable, portable, and governed by declared structure rather than accidental implementation detail.

8. What AEL Actually Is

AEL is not merely an application, a code generator, or a project framework. It is better understood as:

A runtime for governed decision-making over structured intent.

That places it closer to a compiler, workflow engine, rules engine, or orchestration runtime than to a normal line-of-business application.

9. Recommended Development Loop

1. Define minimal YAML
2. Run engine
3. Observe output
4. Refine:
   - graph model
   - policy
   - loop
5. Repeat

The design should be shaped by repeated execution, not by speculative expansion. This is especially important given the novelty of the engine concept.

10. Main Risk to Avoid

The biggest near-term risk is over-modeling before execution.

Failure mode: perfect schemas, rich taxonomy, and elegant theory — but no running system.

The “skinny but tall” method is specifically designed to avoid this trap. It forces the architecture to prove itself through execution as early as possible.

11. Immediate Next Steps

  1. Create main.py and project.yaml.
  2. Implement YAML loading.
  3. Construct a graph with two nodes and one possible edge type.
  4. Implement one policy: “No orphaned code.”
  5. Run the engine and emit a decision or task.
  6. Wrap the result in Docker.

At that point, the engine will exist in a real, end-to-end sense, even if its scope is extremely narrow.

Anchor question: “Does this run end-to-end yet?” If the answer is no, avoid expanding breadth.
