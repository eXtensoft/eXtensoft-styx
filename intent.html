Overview

What MCP-DB is, why it exists, and how it fits the broader design model.

What is MCP-DB?

Most database tools answer operational questions: Is the service up? How many rows are in that table? MCP-DB asks a different kind of question — the kind a senior architect asks when handed an unfamiliar legacy system: What does this database actually reveal about the business domain? Where does the physical structure betray the logical intent?

MCP-DB is a Model Context Protocol server that gives LLMs direct, structured access to relational database metadata across any supported engine. It speaks the language every major database understands — DDL catalogs, statistical profiles, file capacity — and translates that raw information into a normalized, dialect-neutral payload that an evaluating LLM can reason over without needing to understand engine-specific quirks.

The Problem It Solves

Brownfield systems accumulate a particular kind of debt that code metrics cannot capture: the gap between how a database was physically built and what the business actually needed. Tables named after implementation decisions rather than domain concepts. Foreign keys enforcing relationships the domain abandoned a decade ago. Columns with null rates above 90% that no one dare remove. This is physical–logical impedance mismatch — and it is invisible to tools that only report structure without context.

MCP-DB surfaces this mismatch by pairing three independently-queryable metadata tiers, each with its own volatility and cost profile:

  • Schema — LOW volatility. The shape of the data: DDL, columns, constraints, relationships, stored procs.
  • Profile — MEDIUM volatility. The behaviour of the data: row counts, null rates, cardinality, value distributions.
  • Capacity — HIGH volatility. The health of the deployment: file sizes, growth settings, auto-grow history.

How It Works

An LLM connects to MCP-DB via stdio transport and calls one of four tools, supplying a connection string and dialect identifier for the target database. The request flows through a hexagonal architecture:

  1. A Zod-validated McpServerAdapter receives the JSON-RPC call
  2. MetadataService resolves the correct IDatabaseStrategy from its registry using the dialect discriminator
  3. The adapter executes dialect-specific catalog SQL, maps native types through a NormalizedType vocabulary, and returns a clean structured payload
  4. A short-lived SHA-256-keyed TTL cache absorbs burst reads from concurrent evaluators

No credentials are stored server-side. The caller owns its secrets and supplies them on every call.

The Bigger Picture

MCP-DB is the AS-IS interrogation layer in the Three-Ring Design Model.

It feeds a council of LLM evaluators — the ensemble — which extracts the technology-neutral domain model (WHAT-IT-IS) that the database was always meant to represent. A Mediator reconciles their independent findings; a human expert (the Grey Beard) holds final authority.

The result is not a migration plan. It is a clarity instrument — making the gap between what was built and what was intended explicit, auditable, and correctable.

Architecture Decisions

13 pivotal decisions — each with context, selection rationale, and ramifications.

ADR 01 Technology

Implementation Language

Deep .NET experience, but the target ecosystem is the MCP Server SDK — a Node-first world.

Selected TypeScript

Opens the door to the official @modelcontextprotocol/sdk. Closes the door on reusing existing C# DLLs without a bridge.

ADR 02 Pattern

Architectural Pattern: Hexagonal

Need to support multiple DB dialects and multiple UI heads (MCP Server & Electron) from a single core.

Selected Hexagonal (Ports & Adapters)

Strictly decouples "Database Interrogation Logic" from "MCP Protocol." Stdio can be swapped for IPC (Electron) or REST without touching the SQL strategies.

ADR 03 Packaging

Bootstrapper: Manual Composition Root

NestJS is familiar to ASP.NET developers (decorators) but carries high overhead that hides the simple Stdio lifecycle.

Selected Manual Root (main.ts)

Retains a "Pure Node" footprint. No reflection/decorator performance hit. Easy to debug the startup sequence in a restricted environment.

ADR 04 Strategy Asset Management

SQL Externalization: .sql Files via QueryLoader

Dialect-specific "Base" queries must be managed independently — syntax highlighting and isolated testing are requirements.

Selected Strategy-specific resource folders

Enables SQL syntax highlighting and independent query testing. Precludes heavy ORMs (EF/Prisma) in favor of raw performance and metadata precision.

ADR 05 Environment

Deployment: In-House / Air-Gapped

No external registry or cloud access. Running on high-spec local hardware with intentional network isolation.

Selected Local Stdio + Offline pnpm Workspace

Dictates flattened Zod schemas for local LLMs (Ollama) that have lower reasoning density than Claude. Requires manual caching of Node binary dependencies.

ADR 06 Resource Management

Connection Lifecycle: Lazy-Initialized Pooling

The server supports many RDBMS types, but only one is used per session. Static boot-time connections are wasteful and fragile.

Selected Lazy Singleton Pool per Strategy

The system is "self-healing." If an in-house DB goes offline and recovers, the adapter reconnects automatically on the next tool call — no restart required.

ADR 07 Security

Secret Management: Local Environment Variables

Cloud API keys (Claude/OpenAI) and local DB credentials must never be committed to source control.

Selected .env files for LLM keys only — see ADR 16 for DB credentials

LLM API keys stay in .env (git-ignored). Database connection strings are not stored on the server — they are supplied per-call by the consumer. The server holds no DB credentials at rest.

ADR 08 Packaging

Build Output: ESM via pnpm Workspaces

The shared core must be compatible with both the modern MCP SDK and the Electron Main process.

Selected NodeNext / ESM

Requires explicit file extensions on all imports (e.g., ./file.js). Provides the most future-proof path for both Node and Electron targets.

ADR 09 Packaging

Consumer Pattern: Local Monorepo

The core (db-core) must be shared between the MCP Server and a Desktop Tool (Electron) without publishing to any registry.

Selected Symlinked Workspaces (workspace:*)

Enables "live editing" — changes in the core are immediately reflected in both consumers. Precludes versioning overhead during greenfield development.

ADR 10 Connectivity

LLM Agnostic Bridge: Stdio Transport

Must support both local LLMs (Ollama) and cloud LLMs (Claude/OpenAI) while keeping all code and data strictly in-house.

Selected Stdio Transport exclusively

Maintains maximum privacy. Cloud LLMs interact only through a local host (e.g., Claude Desktop). No inbound ports are opened; the server remains invisible to the internet.

ADR 11 Technology

Module System: Pure ES Modules

Modern Node.js requirements for MCP SDKs and future-proofing the shared core library demand a clear module stance.

Selected "type": "module" + NodeNext resolution

Requires strict ESM syntax (file extensions in imports). Ensures seamless integration with the official MCP SDK, which is increasingly ESM-first.

ADR 12 Strategic Intent

Core Purpose: Domain-Physical Fitness Evaluation

This is not a generic DB admin tool. It is designed to expose the "friction" between DB-centric E/R models and domain-centric designs.

Selected Normalized Metadata Enrichment

Every base query must return heuristics (size, uniqueness, constraints) rather than just names. Output is optimized for LLM structural auditing against a companion MCP-FS context.

ADR 16 Security

Credential Model: Per-Request Connection Strings

The server must support multiple callers connecting to different databases without storing any credentials at rest. The original single-connection .env model is too rigid for a multi-consumer interrogation tool.

Selected Connection string passed by caller on every invocation

The server is stateless with respect to credentials. Every tool call carries a connectionString + dialect discriminator. Ad hoc queries follow the same model. Builders must enforce read-only access at the DB permission level, not in application code.

ADR 21 Reference

Connection String Formats per Dialect

IDatabaseStrategy and ConnectionInput accept an opaque connectionString: string. Each adapter expects a specific format. Callers must supply the correct format for the target dialect — the server does no format translation.

Formats ADO.NET · PostgreSQL URL · Oracle EZConnect · MySQL URL · DB2 DSN

sqlserver — ADO.NET key=value:
Server=host,1433;Database=db;User Id=u;Password=p;TrustServerCertificate=True

postgres — RFC 3986 URL:
postgresql://user:pass@host:5432/dbname

oracle — EZConnect or TNS descriptor:
user/pass@host:1521/service_name

mysql — RFC 3986 URL (mysql2 driver):
mysql://user:pass@host:3306/dbname

db2 — ibm_db DSN key=value:
DATABASE=db;HOSTNAME=host;PORT=50000;PROTOCOL=TCPIP;UID=u;PWD=p

Builder note: validate format in the adapter’s normalise() helper before attempting a connection. A malformed string should throw AdapterException { kind: 'validation' } before any socket is opened.

ADR 22 Infrastructure

SQL Script Loader: ESM import.meta.url Pattern

In ESM / NodeNext, .sql files cannot be statically imported. Paths resolved via process.cwd() or a hardcoded base break when the process is launched from a different working directory (MCP host, test runner, etc.).

Selected loadScript(dir, filename) with import.meta.url-relative resolution

loadScript.ts at src/adapters/secondary/ exposes a single async function. Each adapter’s queries.ts derives scriptDir via fileURLToPath(new URL('../scripts/', import.meta.url)) — anchored to the source file’s location, immune to cwd and symlinks. Each Queries entry is a thunk (() => Promise<string>) for lazy loading. loadScript does not cache internally — result caching is handled at the MetadataCache level. Missing scripts surface as AdapterException { kind: 'validation' } (a deployment configuration error).

ADR 19 Architecture

MetadataService: Strategy Registry & Adapter Lifecycle

The hexagon needs a single coordination point that resolves the correct database adapter at runtime, integrates the cache layer, and owns adapter lifecycle — without coupling core logic to any specific driver or MCP protocol detail.

Selected Registry Map<dialect, IDatabaseStrategy> injected at composition root

Strategy instances are long-lived singletons registered in main.ts. Each adapter manages its own internal connection pool keyed by connection string; no adapter holds an idle connection. An unknown dialect throws AdapterException { kind: 'validation' } before any DB call is attempted. dispose() uses Promise.allSettled so one failing adapter cannot prevent others from cleaning up. executeQuery bypasses the cache entirely — ad hoc results are caller-defined and unpredictable.

ADR 20 Performance

MetadataCache: Keyed TTL Cache with Per-Tier TTLs

Multiple evaluators may call the same tool against the same database within seconds of each other. Re-running expensive catalog queries or full-table profile scans for each caller is wasteful and slow. A shared in-process cache absorbs burst reads.

Selected In-process Map with SHA-256 keys, lazy TTL eviction, LRU overflow

TTLs mirror volatility tiers: SCHEMA 1 hour, PROFILE 15 minutes, CAPACITY 5 minutes (set to 0 to disable). Keys are SHA-256 digests of [tier, dialect, connectionString, database?, tables?] — credentials never appear as plain-text map keys. Eviction is lazy (on read) plus insertion-order LRU overflow at maxEntries (default 256). invalidate(key) supports targeted eviction after a known DDL change. Scoped to the current process — if mcp-db is ever deployed multi-instance, replace with a shared store.

ADR 18 Interface Design

MCP Tool Surface: Four-Tool Design

The LLM caller needs a complete but minimal surface to interrogate any supported database. Too many tools create ambiguity; too few collapse distinct volatility concerns into a single call, forcing callers to overfetch or re-call to re-profile.

Selected Four tools — one per IDatabaseStrategy method

Tools: get_schema_metadata (LOW), get_data_profile (MEDIUM), get_capacity_snapshot (HIGH), execute_query (ad hoc). All share a ConnectionInput base (dialect + connectionString) per ADR 16. Zod validates at the port entry — ZodError is immediately wrapped as AdapterException { kind: 'validation' }. McpServerAdapter is the sole catch point for all errors; AdapterException.error.message is returned as isError: true MCP content — the cause field is never forwarded. Successful results are JSON-serialised as a single text content block.

ADR 17 Error Handling

Port Error Contract: AdapterError Taxonomy

Raw driver exceptions (mssql, pg, oracledb) must not cross hexagonal port boundaries in their native form — each driver has its own error shape, codes, and vocabulary. McpServerAdapter needs a stable, driver-neutral error vocabulary to generate safe MCP error responses without leaking internal details.

Selected Discriminated union AdapterError + AdapterException wrapper

Six error kinds cover all failure modes at the boundary: connection, not_found, permission, query, timeout, validation, unknown. The message field is safe for LLM consumption. The cause field preserves the original driver error for local logging and is never forwarded to the caller. Core services do not catch AdapterException — they let it propagate to the primary adapter boundary. Each adapter implements a private normalise() helper mapping driver-specific codes to AdapterErrorKind.

ADR 13 Philosophy

Evaluation Model: Pluralistic Consensus

Single-perspective analysis is prone to bias and hallucination. Fitness evaluation requires independent "eyes" to correlate findings.

Selected Strategy Pattern for Evaluators[]

Must support "Blind Audits" — N evaluators process the same MCP-DB/FS context without seeing each other's work. A Mediator stage identifies the Intersection (Common Truth) and the Delta (Divergent Opinions).

ADR 14 Philosophy

Design Documentation Format

Design artifacts must be consumable by LLMs and humans alike, validatable by machines, and renderable into multiple output formats. A code-first format is too narrow; a document-first format must be the source of truth.

Selected YAML · JSON Schema · Transform Pipeline

TypeScript interfaces and code contracts become derived output from YAML design documents — not the source of truth. A core vocabulary of primitives (ADR, DataContract, Role, Principle) is extended per domain (DataStorage, AppDesign, WebApp) and rendered to any target format without modifying the source.

ADR 15 Philosophy

Three-Ring Design Model

Brownfield migrations fail when AS-WILL-BE is designed against the AS-IS implementation rather than the underlying domain model — inheriting legacy accidents alongside legacy intent.

Selected AS-IS → WHAT-IT-IS → AS-WILL-BE

MCP-DB and MCP-FS interrogate the AS-IS. The Evaluator ensemble extracts the technology-neutral WHAT-IT-IS domain model. AS-WILL-BE is then designed against that clean model — making fitness friction explicit and correctable rather than inherited. A ColdFusion system and a .NET system can share the same WHAT-IT-IS ring; the delta between them becomes auditable.

04

Ensemble Design: The Fitness Audit

Using MCP-DB and MCP-FS together to evaluate physical vs. logical impedance mismatch.

The audit is not a single-model pass. It is a council — multiple independent LLM instances each ingest the same raw metadata and domain context in isolation, then a Mediator correlates their findings. A human expert ("the Grey Beard") holds final authority on the verdict.

🔍
The Evaluators
N independent LLM instances

Multiple independent LLM instances (LLaMA 3, Command R, etc.) ingest raw metadata from MCP-DB and source code from MCP-FS. They identify "Friction Points" where the DB structure contradicts or hinders the Domain Intent.

⚖️
The Mediator
Specialized LLM role

Tasked with Consensus Building. Does not look at the DB directly — it reads the N reports from the Evaluators. It highlights universal truths and identifies where the experts disagree.

🧔
The Grey Beard
Human-in-the-Loop (HITL)

Reviews the mediated report. Evaluates the "Quality of the Evaluation" itself. Provides the final verdict: Refactor, Accept, or Re-audit.
