Adaptive Execution Loop (AEL)

Version 3 — Engineering-spec style position paper with formal definitions and system-oriented structure.

Formal statement: AEL is a stateful, policy-governed software engineering system in which artifacts evolve through a continuous loop that observes state, evaluates admissibility, determines action, executes change, and records resulting evidence.

1. Scope

This document defines AEL at a system level. It does not prescribe a particular implementation language, repository model, storage technology, agent vendor, or SDLC tooling stack. Its purpose is to define the minimum conceptual structure required for an engine capable of governed, adaptive, closed-loop software execution.

2. Problem Definition

Software lifecycle failure commonly arises from one or more of the following conditions:

  • requirements that remain ambiguous or untested until late,
  • weak traceability between upstream and downstream artifacts,
  • slow or sparse feedback between implementation and stakeholder evaluation,
  • human governance that is inconsistent, delayed, or non-machine-readable,
  • lack of structured means to revise upstream artifacts when downstream evidence invalidates them.

AEL is intended to address these conditions by replacing loosely coordinated progression with stateful loop execution under policy and evidence constraints.

3. System Model

System S = (A, P, E, G, D, X, L)

Where:
A = set of artifacts
P = set of policies
E = set of evidence
G = set of gates
D = set of decisions
X = set of executable actions
L = loop function

The engine operates over a current system state S_t and produces a successor state S_(t+1).

4. Primitive Definitions

4.1 Artifact

Artifact:
  id: unique identifier
  type: artifact classification
  state: lifecycle state
  version: revision indicator
  content: structured or unstructured payload
  metadata: auxiliary attributes

4.2 Policy

Policy:
  id: unique identifier
  scope: applicable artifact types or transitions
  condition: machine-evaluable rule
  severity: advisory | blocking
  outcome: pass | fail | unknown

4.3 Evidence

Evidence:
  id: unique identifier
  type: evidence classification
  subject: referenced artifact or transition
  result: structured finding
  timestamp: observed time
  provenance: source of generation or validation

4.4 Gate

Gate:
  id: unique identifier
  required_policies: policy set
  required_evidence: evidence set
  condition: admissibility function over current state

4.5 Decision

Decision:
  id: unique identifier
  trigger: originating event or condition
  target: artifact(s), transition, or gate
  disposition: advance | revise_upstream | regenerate_downstream | retry | escalate | hold

5. Loop Definition

Loop(S_t):
  O_t = observe(S_t)
  V_t = evaluate(O_t, P, G)
  D_t = decide(V_t)
  X_t = act(D_t)
  S_(t+1) = update(S_t, X_t)
  record(S_t, O_t, V_t, D_t, X_t, S_(t+1))
  return S_(t+1)

The loop is the control mechanism of the lifecycle. Phases may contextualize activity but do not replace loop control.

6. Function Semantics

6.1 Observe

Collect current artifact states, evidence states, gate status, operational results, and external feedback relevant to admissibility or correction.

6.2 Evaluate

Assess traceability completeness, policy compliance, artifact validity, success measures, and gate satisfaction. Evaluation should be capable of producing both deterministic findings and unresolved conditions requiring escalation.

6.3 Decide

Select a disposition. A disposition may advance the system, direct revision, request regeneration, block further progression, or escalate to human judgment.

6.4 Act

Invoke agents, modify artifacts, generate artifacts, execute tests, create demonstrations, or create decision tasks. The engine may use one or more external agent systems, but orchestration remains internal to AEL.

6.5 Record

Persist loop telemetry sufficient for replay, audit, and continuous improvement.

7. Upstream Correction Model

AEL explicitly permits correction of upstream artifacts when downstream evidence invalidates them. This is a core structural property rather than an exception condition.

Examples:
If demo_feedback invalidates workflow assumption
  -> revise use case and possibly requirement

If test failure reveals undefined business rule
  -> revise requirement or policy definition

If policy violation reveals architectural inconsistency
  -> revise design or implementation artifact

Upstream correction prevents the engine from falsely treating early artifacts as immutable truth.

8. Traceability Requirement

AEL assumes that artifacts participate in a graph of trace links. At minimum, the engine should be able to relate:

  • intent to requirements,
  • requirements to stories and use cases,
  • stories and use cases to designs and behaviors,
  • designs and behaviors to code and tests,
  • tests, demos, and approvals to evidence and gate satisfaction.

Broken traceability degrades governance quality and weakens confidence in decisions.

9. Admissibility and Gates

Advancement under AEL is admissibility-based, not assumption-based. A gate should be satisfied only when the required evidence and policies jointly support progression.

BuildGate(S) = true iff:
  required design artifacts exist
  blocking policies pass
  required evidence exists
  unresolved severity threshold is acceptable

This makes advancement a computable proposition rather than an informal judgment.

10. Human Escalation Boundary

The engine must distinguish between what can be machine-evaluated and what requires human judgment. Escalation is appropriate where policy is incomplete, competing objectives are unresolved, risk acceptance must be explicit, or ethical, legal, or strategic decisions are present.

Rule: humans should be inserted deliberately where judgment is necessary, not because the lifecycle lacks structure.

12. Comparison with Existing Paradigms

Paradigm Primary strength Relative gap addressed by AEL
Waterfall Planning clarity Insufficient adaptability under changing understanding
Agile Iteration and responsiveness Often lacks formal machine-readable governance
DevOps Automation of build-test-deploy operations Typically does not govern upstream intent and requirement correction
AEL Policy-governed adaptive execution across the artifact graph Requires formalized artifacts, rules, and evidence discipline

13. System Properties

  • Stateful: decisions occur against explicit system state.
  • Adaptive: feedback can trigger both upstream and downstream change.
  • Governed: policy and evidence constrain admissible progression.
  • Traceable: artifact relationships remain part of the decision substrate.
  • Extensible: external agent systems may be incorporated without changing the governing core.

14. Final Position

AEL should be treated as a candidate engineering model for governed agentic execution. Its central claim is that software delivery improves when lifecycle control is organized around a formal loop that continuously evaluates state, corrects misalignment, and records evidence. In this model, the engine governs convergence.
