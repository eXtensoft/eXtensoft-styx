Adaptive Execution Loop (AEL)

First-pass working document covering: (1) artifact graph model, (2) YAML DSL for projects, and (3) engine runtime / orchestration model.

Intent: move from position-paper language toward a machine-operable model. This document is deliberately technology-agnostic and is intended to define the governing core rather than a vendor-specific implementation.

0. Guiding Premise

The AEL engine should not begin with code generation. It should begin with a governed model of reality: what artifacts exist, how they relate, what constraints apply, what evidence is required, and what actions are admissible. The engine then reasons and acts against that model.

Accordingly, this first-pass document is organized into three layers:

  1. Artifact Graph — the persistent model of entities and traceability.
  2. Project YAML DSL — the machine-readable declaration layer for projects and governance inputs.
  3. Engine Runtime — the operating system that executes the Adaptive Execution Loop.

1. Artifact Graph Model

The artifact graph is the substrate of the engine. Artifacts are nodes. Relationships are edges. Policies and gates operate over the graph. Evidence strengthens or weakens confidence in specific subgraphs. Decisions and actions transform graph state.

1.1 Node Classes

Node class Purpose Examples
Intent Declares desired outcome or mission of the project or subproject Business objective, campaign goal, capability objective
Requirement Formal statement of functional or non-functional need Performance requirement, access-control rule
UserStory Actor-centered expression of desired behavior As a contributor, I want to submit content
UseCase Structured interaction flow Author content, route for approval
Policy Machine-evaluable governance rule All external interfaces require contract tests
Risk Known threat, uncertainty, or hazard to success Ambiguous data ownership, latency risk
Decision Architectural or governance choice Use event-driven integration, adopt approval gate
Design Abstract or concrete realization model Context model, sequence, component boundary
Component Named logical unit of implementation Catalog service, approval engine, UI shell
CodeUnit Executable implementation artifact Module, service, class, function
Test Verification artifact Unit test, contract test, scenario test
Evidence Observed proof about an artifact or state transition Passed suite, approval record, metric output
Gate Admissibility boundary Architecture gate, build gate, release gate
AgentCapability Declares an agent’s available competencies Generate code, critique design, produce tests
Task Directed work item for the engine or an agent Implement story, reconcile trace gap, generate demo

1.2 Edge Classes

Edge Meaning
refinesOne node is a more detailed version of another
realizesA downstream node implements or expresses an upstream node
constrainsA policy, requirement, or decision limits permissible form or behavior
verifiesA test or evidence node provides verification for another node
depends_onA node requires another node to be meaningful or admissible
risksA risk node threatens or weakens another node or transition
satisfies_gateEvidence contributes to a gate being passable
assigned_toA task is assigned to a human or agent capability
derived_fromA node was synthesized from one or more precursor nodes
supersedesA newer node version replaces an older one semantically

1.3 Canonical Trace Chain

Intent
  -> Requirement
  -> UserStory / UseCase
  -> Design
  -> Component
  -> CodeUnit
  -> Test
  -> Evidence
  -> Gate

This chain is not always strictly linear, but it defines the minimum traceability expectation. The engine should be able to traverse it in both directions.

{
  "id": "uuid",
  "edge_type": "realizes",
  "from_node_id": "REQ-123",
  "to_node_id": "DESIGN-456",
  "metadata": {
    "confidence": 0.91,
    "created_by": "engine"
  }
}

1.6 Graph Constraints

  • A CodeUnit should not exist without at least one upstream Design, Requirement, or UserStory trace, unless marked exploratory.
  • A Gate should not pass without required Evidence relationships.
  • A blocking Policy violation should create a graph condition that prevents advancement.
  • A superseded node should remain in the graph for history but no longer be treated as the active node unless explicitly referenced.
  • Orphaned downstream artifacts should be surfaced as traceability defects.
Design principle: the graph should preserve history. AEL is not merely about the latest state; it is also about why the system arrived there.

2. Project YAML DSL

The YAML DSL is not the engine itself. It is the declarative surface by which a project, its governance rules, and its initial artifact set are described to the engine. It should be readable by humans, writable by disciplined teams, and translatable into the artifact graph.

2.1 DSL Objectives

  • Declare project identity and intent.
  • Declare actors, artifact definitions, and lifecycle semantics.
  • Declare policies, gates, and success measures.
  • Declare agent capabilities and permitted actions.
  • Allow import or embedding of requirements, stories, use cases, diagrams, and standards.

2.3 Example Project YAML

project:
  id: cms-001
  name: "Content Governance Pilot"
  domain: "content-management"
  description: "Pilot project for governed content authoring and approval"
  owner_role: "ProductOwner"

intent:
  outcomes:
    - "Enable governed content creation and approval"
    - "Reduce cycle time from draft to publish"
    - "Preserve traceability from request through publication"
  constraints:
    - "Human approval required for legal-sensitive content"
    - "All publish actions must have approval evidence"

actors:
  - id: contributor
    type: human_role
  - id: editor
    type: human_role
  - id: approver
    type: human_role
  - id: engine
    type: system
  - id: code_agent
    type: agent
  - id: design_agent
    type: agent

lifecycle:
  phases:
    - inception
    - elaboration
    - construction
    - transition
  states:
    artifact: [draft, proposed, approved, implemented, verified, superseded]
    task: [queued, executing, blocked, complete, escalated]

artifacts:
  requirements:
    - id: REQ-CMS-001
      kind: functional
      text: "The system shall allow contributors to author and submit content."
  user_stories:
    - id: US-CMS-001
      text: "As a contributor, I want to author and submit new content."
  use_cases:
    - id: UC-CMS-001
      actors: [contributor, system]
      steps:
        - "Contributor opens authoring view."
        - "System presents content-type specific form."
        - "Contributor saves draft or submits."

traceability:
  links:
    - from: REQ-CMS-001
      to: US-CMS-001
      type: refines
    - from: US-CMS-001
      to: UC-CMS-001
      type: realizes

policies:
  - id: POL-001
    scope: [publish_action]
    severity: blocking
    rule: "publish requires approval_evidence == true"
  - id: POL-002
    scope: [code_unit]
    severity: blocking
    rule: "all external interfaces require contract_test"

gates:
  - id: GATE-ARCH-001
    type: architecture
    requires:
      policies: [POL-002]
      evidence: [design_review]
  - id: GATE-REL-001
    type: release
    requires:
      policies: [POL-001, POL-002]
      evidence: [acceptance_test, approval_evidence]

success_metrics:
  - id: METRIC-001
    name: "draft_to_publish_cycle_time"
    target: "<= 2 days"
  - id: METRIC-002
    name: "traceability_coverage"
    target: "100%"

agents:
  - id: code_agent
    capabilities: [generate_code, refactor_code, generate_tests]
  - id: design_agent
    capabilities: [generate_design, critique_design, detect_trace_gaps]

tasks:
  generation_rules:
    - trigger: "new_requirement"
      action: "generate_candidate_story_and_use_case"
    - trigger: "trace_gap_detected"
      action: "create_reconciliation_task"

runtime:
  escalation:
    when:
      - "blocking_policy_violation unresolved > 1 loop"
      - "ambiguous requirement confidence < 0.70"
  logging:
    level: detailed

imports:
  - "./requirements.yaml"
  - "./policies/security.yaml"
  - "./standards/coding.yaml"

2.4 DSL Design Notes

  • The DSL should declare what is true, what is allowed, and what is required, not procedural details that belong in runtime internals.
  • It should support modular imports so teams can compose reusable policy packs, domain packs, and standards packs.
  • It should remain vendor-neutral: agents are declared by capability, not by brand name.
  • It should be convertible into graph nodes and edges with minimal ambiguity.

2.5 Proposed Schema Buckets

Declarative buckets

  • Project identity
  • Intent and constraints
  • Actors and roles
  • Artifact declarations
  • Policies and gates
  • Metrics and success targets

Operational buckets

  • Agent capability map
  • Task generation rules
  • Escalation rules
  • Logging configuration
  • Import paths and package references

3. Engine Runtime / Orchestration Layer

The runtime is the active operating core of AEL. It observes the graph, evaluates admissibility, makes decisions, dispatches work, updates state, and records telemetry. The runtime should treat external agents as pluggable executors, not as the system of record.

3.1 Runtime Responsibilities

  • Maintain current graph state and version history.
  • Detect triggers and loop-entry conditions.
  • Evaluate policies, evidence, gates, and traceability.
  • Synthesize or queue tasks.
  • Route tasks to appropriate agent capabilities or humans.
  • Accept outputs, validate them, and merge them into graph state.
  • Persist telemetry and decision history.

3.2 Canonical Loop

Loop(S_t):
  O_t = observe(S_t)
  V_t = evaluate(O_t)
  D_t = decide(V_t)
  X_t = dispatch_and_act(D_t)
  S_(t+1) = update_graph(S_t, X_t)
  T_t = record_telemetry(S_t, O_t, V_t, D_t, X_t, S_(t+1))
  return S_(t+1)

3.3 Observe

Observe should gather:

  • new or modified artifacts,
  • missing trace links,
  • policy failures,
  • test outcomes,
  • demo outcomes,
  • human feedback,
  • unresolved tasks and escalations.

3.4 Evaluate

Evaluate should answer:

  • What changed?
  • What nodes or edges are now invalid, incomplete, or superseded?
  • What policy checks pass or fail?
  • What gates have become passable or blocked?
  • Does new evidence suggest upstream correction?

3.5 Decide

Decision outcomes should be finite and explicit. Example dispositions:

Disposition Meaning
advanceAdmissibility satisfied; proceed to next state or gate
revise_upstreamEvidence indicates issue in requirement, story, use case, or design
regenerate_downstreamDownstream realization should be rebuilt from updated upstream artifacts
retryRepeat the same class of action with revised context
holdWait for dependency or prerequisite evidence
escalateRequire human judgment or exception handling

3.6 Dispatch and Agent Interface

The runtime should not directly encode vendor specifics. It should dispatch tasks to a generic capability interface.

AgentRequest:
  task_id
  capability_required
  input_artifacts
  governing_constraints
  success_conditions
  output_contract

Example capabilities:

  • generate_design
  • critique_design
  • generate_code
  • generate_tests
  • explain_trace_gap
  • summarize_feedback
  • produce_demo_artifact

3.7 Merge and Validation

No agent output should be treated as truth on arrival. The runtime should validate outputs before graph merge:

  • Is the output well-formed?
  • Does it satisfy the requested output contract?
  • Does it preserve or strengthen traceability?
  • Does it violate any blocking policies?
  • Does it supersede or fork an existing node?

3.8 Telemetry Model

LoopLog:
  loop_id
  trigger
  nodes_in_scope
  edges_in_scope
  policy_results
  gate_results
  decision
  tasks_generated
  agent_invocations
  outputs_received
  merge_results
  escalation_events
  latency
  successor_state_id

Suggested derived metrics:

  • mean time from requirement creation to first demo,
  • traceability coverage ratio,
  • upstream correction frequency,
  • policy-violation recurrence rate,
  • human-escalation rate,
  • agent task success rate.

3.9 Escalation Logic

The runtime should escalate deliberately. Candidate escalation conditions:

  • blocking policy violation persists beyond configured retry threshold,
  • conflicting upstream artifacts cannot be reconciled,
  • confidence in inferred requirement interpretation falls below threshold,
  • human-signoff policy applies to the current gate,
  • risk severity exceeds permitted autonomy boundary.

4. Integration of the Three Layers

Graph

Persistent state and traceability substrate.

YAML DSL

Declarative input and configuration surface.

Runtime

Adaptive Execution Loop that reasons, acts, and records.

Project YAML
  -> parsed into graph nodes and edges
  -> runtime evaluates graph under policy and gate rules
  -> runtime dispatches tasks to agents or humans
  -> outputs are validated and merged back into graph
  -> telemetry is recorded for future improvement

5. Recommended Next Refinements

  1. Typed node taxonomy: finalize the allowed node and edge sets.
  2. Graph invariants: define hard constraints and soft warnings.
  3. YAML schema versioning: introduce explicit versioning and package composition.
  4. Policy expression language: define how rules are formally written and evaluated.
  5. Output contracts: define the required structure of agent outputs.
  6. Escalation classes: distinguish ambiguity, risk, authority, and policy exceptions.
Most important next move: formalize the artifact graph invariants. The graph is the spine of the system. If its semantics remain loose, the rest of the engine will drift.

6. Closing Position

This first-pass model treats AEL as a governed system with three indispensable layers: a graph of artifacts and relationships, a declarative configuration and project language, and a runtime loop that orchestrates adaptation. External AI systems may provide execution power, but the governing intelligence remains with the AEL engine.

That distinction is central. The engine governs. Agents execute.
