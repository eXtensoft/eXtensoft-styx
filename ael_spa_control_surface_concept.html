AEL SPA Control Surface Concept

Working memo on the role of an ES6 SPA, backed by a Node or Python server, as the operator-facing host environment for the Adaptive Execution Loop engine.

Core boundary: the SPA is not the engine. The SPA is the control surface over the engine.

1. The central idea

An ES6-based SPA served by a lightweight Node or Python server can absolutely serve as the host environment for AEL. This is not merely a convenience. It is architecturally desirable because it creates a visible, interactive surface over an otherwise abstract governance runtime.

The SPA should be treated as the operator console of the system: a place to inspect graph state, review policy results, submit structured changes, trigger loop execution, and observe decisions, metrics, and outcomes.

2. Architectural boundary

The boundary must remain strict:

SPA (UI / control surface)
  ↓
Host Server (Node or Python)
  ↓
AEL Engine (graph + policies + decisions + runtime loop)

The SPA can request information, trigger actions, and render results. The engine remains the only source of truth for graph semantics, policy logic, decisions, and admissibility.

Non-negotiable rule: the frontend must not contain engine logic, policy logic, or graph-governance rules.

3. Valid hosting patterns

Pattern A: Embedded engine

Node/Python Host
  ├── serves SPA
  └── runs engine in same process

This is the best starting point. It minimizes complexity and accelerates experimentation.

Pattern B: Split engine service

SPA
  ↓
Host API
  ↓
Engine Service (separate process/container)

This is cleaner architecturally and likely the better long-term direction, but it introduces additional moving parts too early for the first vertical slice.

Recommended path: start with Pattern A, evolve later toward Pattern B when the engine model stabilizes.

4. What the SPA actually is

The SPA is not a generic dashboard. It is not just a data-entry front end. It is an interactive, schema-driven control surface for governed engineering.

Visualization surface

Shows graph state, traceability, violations, and runtime outcomes.

Interaction surface

Allows users to submit structured changes, approve decisions, and trigger loop execution.

Observation surface

Displays metrics, trends, timing, and system behavior over time.

5. Why this is powerful

This approach directly compresses the feedback loop that motivated AEL in the first place. Instead of waiting for long requirement-to-demo cycles, the system can support a much tighter interaction loop:

Edit → Submit → Evaluate → Visualize → Refine → Repeat

That means a user, architect, or operator can observe the consequences of a change far sooner. The act of visualization itself becomes a refinement mechanism, which is entirely consistent with the AEL premise that demonstration is a discovery instrument.

6. Primary SPA responsibilities

  1. Graph visualization
    Display nodes, edges, traceability chains, orphans, and supersession.
  2. Policy result display
    Show passed rules, failed rules, severity, rationale, and affected scope.
  3. Loop triggering
    Allow explicit invocation of loop execution, evaluation, or targeted actions.
  4. Decision display
    Present decisions, generated tasks, escalations, and next recommended steps.
  5. Dynamic form rendering
    Render artifact-editing and artifact-creation forms from JSON Schema or equivalent declarative schemas.
  6. Metrics presentation
    Display loop latency, traceability coverage, violation trends, and other system metrics.

7. Dynamic forms and schemas

Schema-driven interaction is especially important because it preserves extensibility. If artifact types, policy types, and task types are declared through schemas, the UI can render forms dynamically rather than hard-coding every interaction surface.

Schema → Form → Submission → Engine → Updated Graph / Metrics / Decisions

This allows the system to grow without rewriting the UI every time the ontology expands.

Interpretation: schemas become part of the language between the operator and the engine.

8. Metrics as first-class output

Metrics are not an accessory. They are part of the feedback system. The SPA is an ideal place to surface them because it makes the behavior of the engine legible to human operators.

Examples of useful first-pass metrics:

  • traceability coverage ratio,
  • number of orphaned downstream artifacts,
  • policy violations by severity and category,
  • loop execution latency,
  • time from requirement creation to first demo,
  • human escalation rate,
  • rate of upstream correction.

9. Runtime discoverability

The host server should expose the information needed for the SPA to discover the system at runtime. That means the UI should not have to know the world in advance.

Illustrative endpoints:

GET  /schemas
GET  /schema/requirement
GET  /node-types
GET  /policies
GET  /metrics
GET  /graph
POST /run-loop
POST /artifact
POST /apply-decision

The more the system can describe itself at runtime, the more adaptable and reusable the SPA becomes.

10. Minimal first implementation

The first UI should remain extremely small. It does not need to be visually polished. It only needs to prove the interaction pattern.

  1. A button to run the engine loop.
  2. A simple area showing policy violations or decisions.
  3. A single schema-driven form for creating one artifact type, such as a Requirement.

If these three things work together, the UI/engine boundary is real and the concept is operational.

11. Recommended host choice

Either a lightweight Node server or a Python server can act as the host. Given the current direction of AEL, a Python host paired with the Python engine is the shortest path for early experimentation. A Node host remains viable, especially if it fits the developer’s established comfort with ES6-based SPA serving.

The deciding factor is not ideology. It is feedback speed and architectural cleanliness.

12. Final framing

The SPA should be understood as the operator-facing surface of AEL. It gives the engine eyes, hands, and a visible nervous system. The graph remains memory. Policies remain rules. Metrics provide awareness. The host server is the bridge. The engine governs.

Compact metaphor:
Engine = brain
Graph = memory
Policies = rules
SPA = senses + control surface
Metrics = awareness
Schemas = language
