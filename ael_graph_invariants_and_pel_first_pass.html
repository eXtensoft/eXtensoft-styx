Adaptive Execution Loop (AEL)

First-pass working document: graph invariants and policy expression language.

Purpose: define the hard and soft rules that keep the artifact graph coherent, and define a machine-usable way to express governance policies that the engine can evaluate consistently.

1. Why this document matters

The artifact graph gives AEL its structure. The runtime gives it motion. But without graph invariants and a policy language, the system has no disciplined way to decide what is valid, what is incomplete, what is forbidden, and what merely deserves warning.

This document therefore defines:

  1. Graph invariants: conditions that must hold, should hold, or should be surfaced as defects.
  2. Policy Expression Language (PEL): a first-pass declarative language for expressing rules over nodes, edges, states, evidence, gates, metrics, and transitions.
Key principle: if the graph is allowed to become semantically loose, the engine will eventually devolve into a document router instead of a governance system.

2. Invariant classes

Not all invariants should be treated the same. AEL should distinguish between hard failures, blocking governance failures, quality warnings, and historical irregularities.

Invariant class Meaning Typical engine response
Structural hard invariant Graph is malformed or logically invalid Reject merge or halt affected branch of execution
Blocking governance invariant Graph is structurally valid but not admissible for advancement Prevent gate passage; create task or escalate
Quality invariant Graph is admissible but below desired quality threshold Emit warning, optionally generate remediation task
Historical integrity invariant Version/history lineage is incomplete or inconsistent Flag defect; usually non-blocking unless audit-critical

3. First-pass graph invariants

3.1 Node identity invariants

  • Every node must have a globally unique identifier.
  • Every node must declare a valid node_type.
  • Every node must have exactly one active lifecycle state.
  • A node version must be monotonic within its semantic lineage.
  • A superseded node may remain in the graph but must not be treated as the default active node.

3.2 Edge integrity invariants

  • Every edge must connect two existing nodes.
  • Every edge must declare a valid edge_type.
  • Edge direction must be meaningful and consistent for its type.
  • Certain edge types may only connect allowed node-type pairs.
Edge type Allowed examples Disallowed example
realizes Design → Requirement, CodeUnit → Design, Test → UseCase Requirement → CodeUnit, unless explicitly modeled as coarse trace
verifies Test → Requirement, Evidence → Gate Requirement → Test
constrains Policy → Component, Requirement → Design CodeUnit → Policy
supersedes Requirement(v3) → Requirement(v2) Requirement → Design

3.3 Traceability invariants

  • Every non-exploratory CodeUnit should trace to at least one upstream Requirement, UserStory, UseCase, or Design.
  • Every Requirement marked implemented should have at least one downstream realization path.
  • Every Requirement marked verified should have at least one verification path through Test or Evidence.
  • Every gate marked passed should have the evidence required by its gate definition.
  • An orphaned downstream node should be treated as a traceability defect.

3.4 State coherence invariants

  • A node cannot move to verified if it has not first been implemented or otherwise realized.
  • A node cannot move to approved if mandatory review evidence is absent.
  • A node in superseded state cannot be the sole active dependency of a current node without explicit waiver.
  • A task cannot move to complete while a blocking policy attached to its output remains failed.

3.5 History invariants

  • If node A_v3 supersedes A_v2, then semantic lineage must remain queryable.
  • Merge events should preserve authorship, trigger, timestamp, and provenance metadata.
  • Agent-synthesized nodes should record generating inputs and governing constraints.
Practical interpretation: invariants are not only about graph cleanliness. They are about preserving the engine’s ability to make trustworthy decisions.

4. Core graph invariant catalog

The following catalog is a first-pass starting set. IDs are included because these should eventually become first-class machine-evaluable rules.

INV-002 Valid Node Type Structural hard Every node must declare a recognized node type.
INV-003 Valid Edge Endpoints Structural hard Every edge endpoint must reference an existing node.
INV-004 Allowed Edge Pairing Structural hard Each edge type may connect only permitted node-type pairs.

5. Policy Expression Language (PEL) goals

The Policy Expression Language should be readable enough for disciplined humans to author and review, but strict enough that the engine can evaluate it consistently. It should express governance conditions over graph structure, node states, evidence presence, metrics, and transitions.

5.1 PEL design goals

  • Technology-agnostic and engine-centric.
  • Readable in YAML or JSON representation.
  • Capable of expressing both hard constraints and advisory quality rules.
  • Explicit about scope, predicate, severity, and remediation behavior.
  • Composable into reusable policy packs.

5.2 PEL should answer four questions

  1. What does this rule apply to?
  2. What condition is being tested?
  3. What happens if the condition fails?
  4. What evidence or exception can satisfy or waive it?

6. First-pass PEL structure

policy:
  id: POL-001
  name: "No orphaned code"
  kind: invariant | gate_rule | quality_rule | escalation_rule
  scope:
    node_types: [CodeUnit]
    states: [implemented, verified]
  severity: blocking | warning | advisory
  predicate:
    all:
      - has_upstream_path:
          from: self
          to_any: [Requirement, UserStory, UseCase, Design]
          via: [realizes, refines, depends_on]
  on_fail:
    action: block_transition
    create_task: trace_reconciliation
    escalate: false
  waivers:
    allowed: true
    waiver_type: exploratory_code
  metadata:
    owner: governance
    rationale: "Prevents implementation outside declared intent"

The structure above suggests that every policy should have at least these sections:

Identity

  • id
  • name
  • kind

Applicability

  • scope
  • node_types
  • states
  • transitions (optional)

Evaluation

  • predicate
  • all, any, not
  • graph/path/existence/count checks

Response

  • severity
  • on_fail
  • create_task
  • escalate
  • waivers

7. First-pass predicate vocabulary

The predicate language does not need to be Turing-complete. In fact, it should not be. It should be constrained and explicit. A small, powerful vocabulary is preferable.

Predicate primitive Meaning
exists_nodeA node matching criteria exists
exists_edgeAn edge matching criteria exists
has_stateTarget node is in a specific lifecycle state
has_upstream_pathThere exists a valid path from current node to one or more upstream node types
has_downstream_pathThere exists a valid path from current node to one or more downstream node types
count_gteCount of matching items is greater than or equal to a threshold
count_eqCount of matching items equals a threshold
metric_satisfiesA measured value satisfies a comparison
evidence_presentEvidence of a declared type exists
gate_satisfiedA referenced gate has reached passable or passed state
no_blocking_violationsNo blocking policies currently fail in scope
transition_requestedA specified state transition is being attempted

7.1 Boolean composition

predicate:
  all:
    - has_state: { target: self, value: implemented }
    - any:
        - evidence_present: { target: self, type: acceptance_test }
        - evidence_present: { target: self, type: approved_demo }
    - not:
        - exists_node:
            type: Risk
            where:
              severity: critical
              status: open

8. Example policies

8.1 No orphaned code

policy:
  id: POL-TRACE-001
  name: "No orphaned code"
  kind: invariant
  scope:
    node_types: [CodeUnit]
    states: [implemented, verified]
  severity: blocking
  predicate:
    has_upstream_path:
      from: self
      to_any: [Requirement, UserStory, UseCase, Design]
      via: [realizes, refines, depends_on]
  on_fail:
    action: block_transition
    create_task: trace_reconciliation
    escalate: false

8.2 Verified requirement must have test evidence

policy:
  id: POL-VERIFY-001
  name: "Verified requirement has evidence"
  kind: invariant
  scope:
    node_types: [Requirement]
    states: [verified]
  severity: blocking
  predicate:
    any:
      - has_downstream_path:
          from: self
          to_any: [Test]
          via: [verifies, realizes]
      - evidence_present:
          target: self
          type: verification_result
  on_fail:
    action: revert_state
    create_task: missing_verification_repair

8.3 Release gate requires acceptance evidence and no blocking violations

policy:
  id: POL-GATE-REL-001
  name: "Release admissibility"
  kind: gate_rule
  scope:
    node_types: [Gate]
    where:
      gate_type: release
  severity: blocking
  predicate:
    all:
      - evidence_present:
          target: self
          type: acceptance_test
      - evidence_present:
          target: self
          type: approval_evidence
      - no_blocking_violations:
          target: release_scope
  on_fail:
    action: deny_gate_pass
    escalate: true

8.4 Demo latency quality rule

policy:
  id: POL-METRIC-001
  name: "Time to first demo"
  kind: quality_rule
  scope:
    node_types: [Requirement]
    states: [approved, implemented]
  severity: warning
  predicate:
    metric_satisfies:
      metric: time_to_first_demo_hours
      op: lte
      value: 72
  on_fail:
    action: create_task
    create_task: accelerate_demo_feedback

9. Policy kinds

Policy kind Purpose
invariantDefines conditions that should always hold

quality_ruleDefines quality expectations that may not block progression

waiver_ruleDefines conditions under which exceptions are acceptable

10. Waivers and exceptions

An engine without exceptions will be brittle. An engine with uncontrolled exceptions will be meaningless. Therefore waivers must be explicit, typed, and recorded.

waiver:
  id: WAV-001
  applies_to: POL-TRACE-001
  subject: CODE-EXPERIMENT-123
  reason: "exploratory prototype"
  approved_by: architect_role
  expires_at: 2026-12-31
  compensating_controls:
    - "must not be promoted to production branch"
    - "must be reconciled before release gate"

Suggested waiver rules:

  • Waivers should never silently remove history.
  • Waivers should be time-bounded where possible.
  • Waivers should identify compensating controls.
  • Waived conditions should remain queryable and visible to gates.

11. Engine evaluation model for policies

for each policy in applicable_policies(scope):
    result = evaluate(policy.predicate, graph_context)

    if result == fail:
        record_violation(policy)
        apply(policy.on_fail)

    if waiver_exists(policy, subject):
        modify_effect(policy, waiver)

aggregate_results()
update_gate_status()
generate_tasks_or_escalations()

The evaluation system should be deterministic given the same graph state, policy set, and waiver set. That matters for auditability and trust.

12. Recommended invariant-to-policy boundary

One design choice is whether invariants live outside the policy language as engine internals, or whether they are represented in the same language as policies. My recommendation for AEL is:

Engine-native invariants

Use for irreducible structural truths such as valid node identity, valid endpoints, and basic graph shape.

Policy-expressed invariants

Use for governance conditions like traceability, evidence requirements, and transition admissibility.

Quality rules

Use for non-blocking but valuable expectations such as loop speed, demo latency, or warning thresholds.

Reason: not every truth should be configurable. Some conditions should remain constitutive of the engine itself.

13. Recommended next refinement

  1. Finalize the allowed node-type and edge-type matrices.
  2. Separate engine-native invariants from configurable policy-expressed invariants.
  3. Define the exact path semantics for has_upstream_path and has_downstream_path.
  4. Introduce schema versioning for PEL.
  5. Define a compact formal grammar or JSON Schema/YAML schema for PEL documents.
  6. Model gate status as computed state rather than manually asserted state.
  7. Define how metrics are computed and attached to graph context.

14. Closing position

The artifact graph is the spine of AEL. Graph invariants protect its integrity. The policy language gives the engine a controlled way to evaluate admissibility, quality, and escalation. Together, they turn the lifecycle from a set of ideas into a machine-governable system.

Without graph invariants, the engine cannot trust its state. Without a policy language, it cannot explain or consistently enforce its judgments.
